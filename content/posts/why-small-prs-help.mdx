큰 PR은 작성한 사람도 설명하기 어렵고, 리뷰하는 사람도 집중하기 어렵습니다. 그래서 저는 작업을 기능 단위보다 조금 더 작은 **검토 가능한 크기**로 나누는 연습을 하고 있습니다.

## 작은 PR의 장점

- 리뷰 포인트가 명확해집니다.
- 회귀 위험을 좁게 볼 수 있습니다.
- 배포 전 확인 범위가 줄어듭니다.
- 대화가 빨라집니다.

## 나누는 기준

저는 보통 아래 기준을 함께 봅니다.

1. UI 구조 변경
2. 상태 관리 변경
3. API 연동 변경
4. 리팩터링

하나의 PR에 성격이 다른 변경이 너무 많이 섞이면, 의도 전달이 급격히 어려워졌습니다.
